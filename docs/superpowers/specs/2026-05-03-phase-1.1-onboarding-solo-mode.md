# Phase 1.1 — Onboarding & Solo Mode

> 目標：讓沒有伴侶的友人也能開始試用，同時補上設計稿已規劃但未實作的 Onboarding flow。
> 優先級：P1.1（Phase 1 完成後立即接續）

---

## 背景與動機

Phase 1 假設使用者一定是雙人組合。但在友人測試階段，部分使用者可能：

- 還沒有伴侶
- 伴侶不一定願意馬上加入
- 想先自己熟悉介面再邀請對方

設計稿（`Futari · 全部畫面（列印版）.pdf`，02 · Onboarding）在邀請步驟已規劃「稍後再邀請 →」CTA，代表 Solo Mode 是設計師原本就預期的使用情境。

此外，目前 Onboarding flow（sign-in → setup → dashboard）完全沒有品牌設計，與設計稿落差很大，需要補實作。

## 功能範圍

### 1. Onboarding Flow 重新設計

完全重寫 `/sign-in` 與 `/setup` 的 UI，對齊設計稿的三步驟流程。

**Step 1 — 歡迎畫面（`/sign-in`）**

- Futari logo + ふたり 標語
- 副標：「兩個人的日子，可以一起記下來。」
- Google 登入按鈕
- 服務條款 + 隱私權政策連結（小字）
- 背景：品牌色，mobile-first 全螢幕

**Step 2 — 建立群組（`/setup` Step 1）**

- 標題：「幫你們的家計簿取個名字」
- 副標：「之後可以隨時改。簡短一點比較好記。」
- Text input：placeholder 空白，右上角字數計數 `{n}/20`，maxLength=20
- 建議 chips（點擊帶入）：「我們倆」、「○○家」、「日日」、「Home」、「一起」
- CTA：「下一步」（disabled 當 name 為空）

**Step 3 — 邀請對方（`/setup` Step 2）**

群組建立後進入此步驟。

- 標題：「把連結傳給對方」
- 副標：「對方點開後就能加入『{groupName}』。」
- 已加入成員列表：顯示自己（已加入），對方顯示 pending avatar + 「還在等對方加入」
- 邀請連結 row + 「複製」按鈕
- 原生分享列（LINE / 訊息 / 更多）→ Web Share API，不支援時 fallback 只顯示複製
- 底部 CTA：「稍後再邀請 →」→ 進入 Solo Mode，導向 `/dashboard`

### 2. Solo Mode

**定義**：`group.member_b` 為 `null` 的狀態。

#### 2a. 功能限制

| 功能 | Solo 狀態行為 |
|---|---|
| 新增交易 | 分攤選項（全部我的 / 全部對方的 / 平分）**全部隱藏**，固定為「全部我的」 |
| Dashboard balance | 不顯示 balance hero，改顯示邀請 banner（見下） |
| 結算 | 入口隱藏（無對方，無結算意義） |
| 其餘（列表、設定、資產）| 正常運作 |

#### 2b. 邀請 Banner（Dashboard）

Solo 狀態時，Dashboard 頂部取代 BalanceHero，顯示：

```
[avatar placeholder]  還在等對方加入
傳連結邀請他
[傳送邀請]  按鈕
```

點擊「傳送邀請」→ 呼叫 `createInvite` 取得連結 → 觸發 Web Share API 或複製。

#### 2c. 升級為雙人模式

對方接受邀請（`/invite/[token]`），`member_b` 填入後：

- Realtime postgres_changes 觸發 `OikosGroups` 更新
- `RealtimeProvider` event bus 廣播 `group_updated`
- Dashboard layout server re-fetch（`revalidatePath('/dashboard')`）
- Banner 消失，BalanceHero 出現，AddSheet 解鎖分攤選項

#### 2d. 分攤邏輯 in Solo Mode

新增的交易一律 `split_type = 'all_mine'`，`paidBy = viewer.id`。這些記錄在對方加入後**不需要 retroactive 處理**——solo 時期的記錄就是「自己的費用」，語意完整。

## 實作拆解

### Schema / DB

不需要改動。`member_b` 已 nullable，`GroupBalance.balance` 在 solo 狀態下預設 0 也正確。

### 新增 / 修改的檔案

```
app/
  sign-in/
    page.tsx              ← 重寫 UI（品牌設計）
  setup/
    page.tsx              ← 拆成兩步驟 state machine
    SetupForm.tsx         ← 重寫：Step1 建立群組 / Step2 邀請
  (dashboard)/
    dashboard/
      _components/
        SoloBanner.tsx    ← 新增：邀請 banner
        BalanceHero.tsx   ← 修改：solo 狀態下不 render
        AddSheet.tsx      ← 修改：solo 狀態下隱藏分攤選項
      page.tsx            ← 修改：傳入 isSolo prop
    layout.tsx            ← 修改：推導 isSolo，傳入 MemberContext
  _components/
    MemberContext.tsx     ← 修改：加入 isSolo: boolean
```

### MemberContext 修改

```ts
export interface MemberContextValue {
  group: { id: string; name: string }
  viewer: Member
  partner: Member | null
  viewerIsA: boolean
  isSolo: boolean          // ← 新增
}
```

`isSolo = partner === null`，在 `layout.tsx` 推導後傳入。

### AddSheet Solo 邏輯

```tsx
const { isSolo } = useMember()

// solo 時分攤欄位不 render，action 呼叫時強制帶 split_type: 'all_mine'
```

### Web Share API（邀請）

```ts
const share = async (url: string) => {
  if (navigator.share) {
    await navigator.share({ title: 'Futari 邀請', url })
  } else {
    await navigator.clipboard.writeText(url)
    // toast: 已複製
  }
}
```

## 驗收條件

- [ ] 首次登入後看到品牌化歡迎畫面，非裸 Google 按鈕

- [ ] 邀請步驟：複製連結、Web Share API（iOS Safari 驗證）、「稍後再邀請」可跳過
- [ ] Solo Dashboard：不顯示 balance，顯示邀請 banner
- [ ] Solo 新增交易：分攤欄位不出現，記錄為 `all_mine`
- [ ] 對方加入後：banner 消失，分攤解鎖（Realtime 觸發，不需重整）
- [ ] 現有雙人 group 不受影響
